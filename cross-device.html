Why Cross-Device Matters

The trust gap between mobile and desktop

The SOTTO SDK continuously collects 177 of 291 behavioral attributes in real-time. Hardware-backed identity via Secure Enclave and cap_seed makes spoofing computationally infeasible.

💻 Desktop/Browser: The Gap

No hardware-backed identity. Session tokens are stealable. Cookies are replayable. Most desktop auth relies on one-time credentials that don’t adapt to behavioral change.

The 5-Step Flow

1

Phone Authenticated

Continuous 177/291 attribute collection is already running. The device has a hardware-backed trust score maintained by the SOTTO SDK.

2

Browser Session Detected

The BL1NQ browser SDK initializes and begins collecting 120 behavioral signals — keystroke dynamics, mouse curves, canvas fingerprints, and more.

3

Environmental Correlation

Fuzzy matching across shared environmental signals: same WiFi BSSID, IP subnet, Bluetooth beacon proximity, and barometric pressure readings within threshold.

4

Trust Inheritance

When correlation exceeds the confidence threshold, the mobile trust score propagates to the browser session. No user interaction required.

5

Continuous Dual Monitoring

Both behavioral gates run in parallel. An anomaly on either device triggers escalation — step-up authentication, session restriction, or lockout.

Zero-friction handoff: The user simply opens their browser. If their authenticated phone is nearby and environmental signals correlate, trust is inherited automatically. No QR scan, no OTP, no push approval.

Hardware RNG & Entropy Health

Cryptographic integrity starts with provably random key material

Hardware Random Number Generation

Every key pair (Key 1a on mobile, Key 1b on browser) is generated using hardware-backed entropy sources:

  • 🔐 Secure Enclave TRNG (mobile)
  • 🔐 Web Crypto API + OS entropy pool (browser)
  • 🔐 NIST SP 800-90B compliant health tests
  • 🔐 Continuous entropy monitoring & reseeding

Entropy Health Attestation

The SDK continuously monitors entropy source quality and provides real-time health attestation:

● Healthy
TRNG output passing all NIST tests
● Degraded
Fallback to software CSPRNG

If entropy health drops below threshold, the SDK triggers re-keying and alerts the trust engine. Degraded entropy never silently propagates to key material.
